Roadrunner roadmap

General API design

Xodr format is not so human readable. Instead we introduce the intuitive concept of "road profile".

  • To make things simpler, it only supports fixed-width driving lanes. (no sidewalks, no shoulders)
  • Lane section can be simplified as number of lanes & divider width (zero, half-lane, full-lane, etc.)
  • Bi-directional road has 2 road profiles for each lane section; single-dir road has only 1.
  • Ref lines are modeled as defined by xodr, but we can start with line + cubic spline only.

    • xodr exporter

    Before any UI was developed, the only way to visualize stuff is through online openDrive viewer Thus, the first modification to libOpenDRIVE is the ability to export to xodr.

    Road profile -> xodr road converter

    An algorithm that converts the easy-to-understand "length:roadProfile" representation to xodr format that requires

  • Continuous lane border / width change
  • Well-defined linkage (successor/predecessor) for each lane
  • Unit test: 100+ random-generated profiles to be converted.
  • Results are automatically checked for error.

    • Generate junction from incoming road info

    On a high level, a junction is basically a set of connecting roads, plus any traffic rules like signal phase. Given the full detail of incoming & outgoing roads, we have to ensure that:

  • All incoming lanes have at least one successor connecting road
  • No connecting roads cross each other unless they have to.
  • Error out if these conditions cannot be satisfied
    • There's also a set of 100+ cases, each consisting of 3 to 6 incoming roads with random profiles, to test the robustness of the algorithm. Currently, users have no option to modify the junction once it's generated. This is a planned feature for the future.

    Road graphics as QGraphicsItem

    This step marks an important turning point: we can finally view the map right in a Qt graphical interface! With Qt's QGraphicsScene and QGraphicsView, it's pretty easy to get a 2D view of the map going. We just need to flip the Y axis of the view by defining zoom to (1, -1) to overcome the difference in coordinate system.

    Road operations: split, join, reverse

    To make our way towards graphical road editing, we need to move our API to a higher level.

  • Operating on road means operating on both ref line and lane profile.
  • Split arises from the need of forming a junction.
  • Join is essential for extending an existing road.
  • Reverse does nothing visible but change the ref line direction. It's turns the task of joining "A->B and C->B" to "A->B and B->C", which is more straightforward.

    • Extra caution should be given to corner cases, like a ref line is split at a neglectable distance (1mm) from its end. We put a limit on the minimum length of ref line and lane section to avoid numerical issues.

    Draw ref lines

    Since we only allow line + cubic spline for now, the logic to define ref line is simple:

  • 1st click defines the starting point. This can be another road's (existing) end point.
  • 2nd click defines the control point. If both clicks are on existing end points, simply connect them and we are done
  • 3rd click defines the end point, as well as the starting point of the next segment. This can be combined with the 2nd click (into a double-click) to form a straight line.
    • In fact in most cases we are creating quadratic spline rather than cubic, except the former generates weird geometry.

    Cubic spline works fine in most cases including junctions, but it's hard to draw the circular shape of roundabouts and complicated interchanges. Fortunately, OpenDRIVE supports arcs and spirals, marking an important direction of improvement.

    Localization in road coordinate system

    Localization means finding the road at a 2D point (or area), and where are we at inside its local (s, t) coordinate system, i.e. what's the closest point on its ref line, and which lane we are on.

    Of course we can develop a formula or iterative algorithm to calculate this directly, but it's not easy to gain full robustness given the various parametrized curves we are dealing with. Not to mention as such a frequently-called function (at each frame the cursor is moved!), we have to carefully limit its execution/iteration time or the cursor could feel laggy.

    Thanks to the spatial query API by QGraphicsScene, it's much simpler if we deal with this in a discrete manner. We divide each lane into fine-grained rectangle-like polygens, each of which storing its position in local coordinate system. By asking Qt which polygen we are at, we get a rough estimate of our answer, which can be further improved by interpolation.

    This method benefits us a lot when computing the overlapping area of two roads, by collecting all overlapping pieces of polygons and gluing them together (where MultiSegment helper class is used for). It not only provides the intersecting point of two ref lines, but also the entire area where a junction is needed, depending on the local width and curvature of each road.

    Form junctions while drawing roads

    With the localization technique in place, drawing junctions is as simple as 1,2 and 3:

  • Determine the section of roads that overlap as (s1, s2), with some margin left.
  • Cut each road to (0, s1), (s2, Length). If either section vanishes form a T-junction. Abort if both vanish.
  • Call junction generation API defined earlier to calculate connecting roads.

    • Incremental road graphics

    With any of operations we do to roads (split, join, reverse), the road itself changes or vanishes, but there's absolutely no need to re-create its graphical representation (QGraphicsItem), because it's expensive! We can easily afford to ce-create the QGraphicsItems for a 100-meter road, but what if user extends it for 100 times? By the end of day we will repeat the work above for 1+2+...+100 = 5050 times, but only the last 100 count. That's 98% wasted!

    Therefore, we must be more careful about how we handle a road's graphics during those operations. For instance, with a "join B to A" operation we can simply std::move most pieces of B to A, with the exception of the join point where certain length of transition needs to be generated.

    This results in an important property: time complexity of any single edit won't grow linearly with map size. Of course, as a common practice we still have to limit the length of new road created at a time (currently 500 meters) to prevent the UI from temporarily freezing.

    Add road to existing junction

    Convenience feature to turns a T junction into an X junction. Just draw a road that overlaps with any existing connecting road of one or two existing junction(s).

    Save/Load, Undo/Redo with ChangeTracker

    With the number of features (and bugs!) introduced so far, we need something that can easily bring us to an earlier state, either due to a bug that causes crash, or a misclick resulting in some unwanted changes.

    At the end of each successful operation that made change to the map, ChangeTracker records and serializes the actual changes (currently, any added / deleted / modified road & junction xml node) and apply them to the maintained (serializable) xodr map . The changes are also stored in the undo/redo stack, which are then used for undo/redo.

    The existance of continuously-maintained serializable map is a key feature that didn't exist in RoadBuilder. As a bi-directionally convertable twin of what's happening inside the scene, objects are referred to and linked by string/int IDs as opposed to pointers. These are like two sides of a coin:

  • We still need smart pointers to store objects relationship, as they require minimum care of life cycle
  • ID-based linkage is readily serializable and reliably reflects the state of program, making it easy to store and recover.

    • The cost of maintaining this pair of twins, obviously, is that we must always ensure their consistency. That's the purpose of the following feature.

    Verification at every moment

    Unit-testing works well for complex logic blocks whose input can be easily defined and randomized. In addition, we need integration tests that check how those building blocks work together as a whole. However, it's much more costly to maintain hard-coded test cases: as the feature set evolves, so does the expected output of a given input. Updating those expectations requires a lot of manual work.

    Then what about let the program test itself? In the context of a map editor, it translates to "let the program check if what's drawn match what's serialized". The most common check we perform is that, if two lanes are supposed to be predecessor/successor, then they must meet at about the same point (with 1cm tolerance due to numerical inaccuracy). There are quite a few similar checks throughout the "verification" process; any violation will cause an exception to be throw.

    More importantly, this verification is by default performed at the scope of the entire map, at the moment of every change. This makes sure every bug/inconsistency immediately comes to our attention at the earliest moment, and we cannot easily get away without fixing it. As a development-assisting feature, I also decide to include this in the production version, although turned off by default, so that anyone can report bugs and help improve the software if they want to.

    Lane section profile editing widget

    Flexible lane section profile definition is an important feature unique to RoadRunner. Instead of letting user choose from a collection of pre-defined profiles, this widget uses two draggable horizontal bars and four vertical handles to let one decide how many lanes to create on the left and right side, and how much space to leave for the divider.

    The boundary dark/light background is where the ref line lies. While normally the ref line lies right in the middle of section, it's allowed to put more lanes on one side than the other, creating shifting lanes without bending the ref line.

    In additional to manually dragging those bars and handles, one can also pick from previous profile by right-clicking (and dragging) on any existing road.

    Highlight hovering display

    When hovering, a road is highlighted with the following information displayed:

  • Road ID
  • Length along ref line and lane ID
  • Ref line direction

    • To see more details on the console regarding road profile and junction info, press key I.

    If multiple roads are overlapping at the spot, press key A to cycle highlight among them.
